Nathan Bhat's Blog for the course: DH2650 Computer Game Design

2019-10-31

Yesterday we had a long meeting where each group member pitched many different simple concepts that they had prepared before and we decided on one concept to have as our main idea. The concept we went for is a time based 2D platformer game. The player starts off in the game, and can perform different basic actions, then the player can hit a "reset" button that resets the game. When the player resets, we start over from where we started before, but there is a copy of the player doing everything that was done before the reset. The player can reset many times (an exact limit is yet to be decided), causing there to be a bunch of copies of the player running through their different timelines, a la Super Meat Boy when you complete a level and see all of your attempts play out at once. This mechanic is not only visual though, one "run" could involve the player running up and jumping on some spikes, then reseting and waiting until the previous run jumps on the spikes and then using the corpse as a platform to get past the obstacle. A similar concept is the game "Life Goes On" which is on Steam, however in that game time does not reset, there is just a new character that spawns and can use the corpses of previous attempts in various ways to solve the puzzles.

The platform we decided on is PC, and we will use unity to create the game. We have already started working on a basic prototype to see if the mechanic works, both from a technical perspective and a gameplay perspective. We believe this core mechanic will be feasible to create, as we do have experienced devlepors (myself narcissistically included), however if it is fun or not is yet to be seen. We believe it could be fun, but to be sure we are creating a prototype ASAP. One of the challenges we immediately see is Asset creation. None of us are artists (though some of us have some experience in having to make art for games despite lack of skill or knowledge) which means whatever aesthetic we decide on needs to be very simple and easy to use.

We have many new ideas brewing, and written down in some shared documents, so you'll have to check back to see what we decide on next!

2019-11-03

Today I added the base movement for the character. I reused some code from an earlier project I created a couple years ago, with some modifications to make it fit our game more. In this solution I created a method for reading input that was designed for Unity's FixedUpdate (unity natively reads input on the Update call). Movement included moving right and left, jumping and deceleration to make the character feel a little more natural (a little slidy always feels a little better).

I also added a technique that is not super well known but is present in a lot of the best platformer games, where if the player walks off the side of a platform they will still be able to jump for a few frames after this, despite being in the air. This is because players are often not the best at timing a jump perfectly with reaching the edge of a platform, so doing this improves the experience but is rather subtle.

2019-11-05

Today I added a system for Switchables and Switches. A switch is like a button or lever or something else and a Switchable is like a door or laser or something that can switch on and off. My basic implementation let me create a button that when the player or a clone stands on a door will open, and if the player leaves the button the door will close.

I have an idea for a weight scale type switch, where there is an imbalanced scale where you have to use clones to balance it and when that happens a door will open. I think this could be an interesting puzzle.

2019-11-05

I fixed a small bug I found relating to the door so that now you can have a door that is default open and when a button presses the door closes. This could lead to some interesting designs like where you have to close doors to protect the player from turret fire.

2019-11-08

Today I created a system for turning into a solid that the player and clones can jump on. We have ideas for using this to create stairs using clones and other things. At the moment clones when solid freeze in their position, but I think it might be fun to make a solid clone have phsyics, so the player can push solid clones around etc.

2019-11-09

Since our game is intended to be a free roaming metroidvania style game and our core mechanic is resetting time to t0, we needed some sort of checkpoint system. Because of this I thought of the different requirements each checkpoint would need, such as the ability to clear the clones spawning from a checkpoint and stuff like that. I then wrote the code for this system which took longer than expected, but in the end it worked rather nicely.

Our design idea for this is that each "room" is a level/puzzle and at the end/beginning of each room there should be a checkpoint.

I wasn't intending to work on game design today but it is just too fun to resist. We wanted a method for killing our clones and I wound up writing some code for shooting clones. At the moment shooting solid clones does not remove them, and I'm not sure how we'll want that later on. It all works but I think we need to do some design choices to slow our game down a little (like character movement speed, and maybe a little charge animation for the laser). Right now it's very fast and twitchy which does not feel 10/10. I think some simple modifications will add a bunch to the game feel.

2019-11-22

I implemented the balance switch system today. This system is a puzzle where you have to create balance in a weight scale to activate a switchable (such as a laser or a door). This system also allows for a branching structure, so one balance splitter can split off into another balance splitter and you have to solve the whole thing to activate the switchable.
